# Project Structure Review + Optimization Notes

## Current structure assessment

The project is a React + Vite single-page app that still preserves older HTML/vanilla-JS page templates under `src/legacy/`.

Recommended mental model:

```text
src/
  App.jsx                 React shell, hash routing, legacy page mounting
  main.jsx                React entry
  context/CartContext.jsx Shared cart state bridge between React and legacy pages
  pages/CartPage.jsx      New React checkout/cart page
  components/             New React UI pieces for cart and table selector
  i18n/                   Language helpers and translations
  legacy/                 Preserved old page templates and browser globals
  styles/style.css        Main style file
public/images/            Static menu and about images
server.js                 Simple Node backend for local/API usage
data/                     Runtime JSON data generated by backend
```

## Main findings

1. **Do not package `node_modules`, `.vite`, or `dist` in normal source zip files.**
   These folders are machine-generated and can break after moving between Windows, Linux, or AWS build environments. The uploaded zip included `node_modules`, and Vite/Rollup failed until dependencies were reinstalled.

2. **Cart state is in a transition stage.**
   The project has a proper React cart context, but several legacy pages still call global functions. This is acceptable for now because `CartContext` exposes bridge functions to `window`, but the long-term goal should be moving menu/home/dish detail pages into React components.

3. **Dine-in phone logic conflicted with the UI copy.**
   The page said phone was optional, but validation required a 10-digit phone before placing dine-in orders. This has been fixed: dine-in can submit without phone as a temporary/guest order; if a phone is entered, it must be 10 digits.

4. **Server-side order validation matched the old phone-required behavior.**
   The backend now accepts dine-in orders without phone, while still requiring phone for takeout.

5. **Package scripts were made more portable.**
   Scripts now call Vite through `node node_modules/vite/bin/vite.js`, which avoids executable permission issues after zip/unzip.

## Changes made

- Updated `src/pages/CartPage.jsx`
  - Dine-in phone is truly optional.
  - Dine-in orders without phone become `guest` / temporary orders.
  - Dine-in orders with phone become `member` orders.
  - Fixed success page phone display so it no longer renders stray template literal backticks.
  - Removed the `required` attribute from the optional dine-in phone input.

- Updated `server.js`
  - Dine-in phone can be empty or 10 digits.
  - Takeout still requires 10-digit phone.
  - `userType` is now `member` only when a valid phone exists; otherwise it becomes `guest`.

- Updated `src/i18n/translations.js`
  - Chinese and English copy now correctly explains that dine-in phone is optional.

- Updated `package.json`
  - More portable scripts:
    - `npm run dev`
    - `npm run build`
    - `npm run preview`
    - `npm run dev:host`
    - `npm run clean`

- Updated `.gitignore`
  - Added `data/orders.json` because backend runtime orders should not be committed into source history.

## Suggested next optimization stages

### Stage 1: Stable deployment package
Keep this layout, but always distribute without `node_modules`, `.vite`, and `dist`.

Local run flow:

```bash
npm install
npm run build
npm start
```

Development flow:

```bash
npm install
npm run dev
```

### Stage 2: Reduce legacy bridge code
Move these pages from `src/legacy/pages/` into real React pages one by one:

1. `home.js`
2. `menu.js`
3. `dishDetail.js`
4. `member.js`
5. `admin.js`

Do not rewrite everything at once. The current bridge lets the website keep working while each page is migrated.

### Stage 3: Data structure cleanup
Move hardcoded menu data into one canonical file, for example:

```text
src/data/menuItems.js
src/data/menuCategories.js
```

Then make both menu page and cart page consume the same data source. This will prevent price/name mismatches.

### Stage 4: Backend/database upgrade
The current `server.js` JSON-file backend is fine for local testing. Before real production, move orders/reservations to a database such as PostgreSQL, DynamoDB, or another AWS-managed database.
